# Technology Steering — RE-SAP IFRS 16 Addon

## SAP ECC Current Landscape

> **ARCHITECTURAL MANDATE — OPTION B:** The Z addon is the system of record for lease contracts. RE-FX is NOT the primary contract management system. See `.kiro/steering/option-b-target-model.md`.

The target platform is **SAP ECC 6.0**. Key landscape assumptions:

- **The Z addon replaces RE-FX** for all lease contract management, IFRS 16 accounting, and ROU asset processing.
- RE-FX may be installed in the system but is NOT used as a runtime system by the addon (exception: one-time migration read-out if client has existing RE-FX contracts).
- **FI-AA (Asset Accounting)** is the ROU asset management system — the addon creates/manages ROU assets via standard FI-AA BAPIs directly.
- **FI-GL (New GL)** is active — the addon posts all lease accounting documents directly via standard FI BAPIs/FMs.
- Document splitting in New GL may apply — confirm with FI team at T0-04 workshop.
- The system runs on Unicode with support for ABAP 7.4 or higher — syntax and OO features aligned accordingly.
- Transport system (CTS) is active; all Z objects must be assigned to proper transport requests and packages.
- No RE-FX BAPIs are used. All SAP standard integration is via FI-GL BAPIs, FI-AA BAPIs, and standard SAP service interfaces.

## Future S/4HANA Compatibility Goal
The addon must be **S/4HANA migration-aware** even though the execution target is ECC. This means:

- Avoid usage of obsolete or deprecated ABAP statements (no `MOVE`, `COMPUTE`, classic `SELECT *` without explicit field lists).
- Prefer ABAP CDS views over direct table reads for reporting and data extraction where feasible.
- Under Option B, Z tables are self-owned — there is no dependency on RE-FX table structures that may change in S/4. This makes Option B inherently more S/4-compatible than Option A.
- Document every ECC-specific assumption with a flag: `[ECC-SPECIFIC: Review for S/4 migration]`.
- Do not create ALV-only UIs without noting the Fiori alternative for S/4 — even if Fiori is not built for v1.
- FI-GL and FI-AA BAPI wrappers must be behind interface classes to allow S/4 API substitution without changing business logic.

## Z Object Strategy
All custom development must follow the **Z namespace** with the project prefix `ZRE_IFRS16_` (example, to be confirmed with ABAP Architect and transport landscape).

Principles:
- **Encapsulate, do not modify standard.** Use BAdi, user exits, and enhancement frameworks — never modify standard SAP code.
- **Stateless design where possible.** Z functions and classes should be stateless; state is persisted in Z tables.
- **Package structure:** One development package per functional area (calculation engine, data capture, disclosure, etc.) — exact package naming in `.kiro/steering/structure.md`.
- **Reuse over duplication.** Common utilities (logging, error handling, number formatting) go into a shared Z utility class library.

---

## ABAP OO Preference
All new ABAP development uses **Object-Oriented ABAP** exclusively:

- Classes over function modules (exception: wrapper classes over legacy FM calls may be acceptable for standard SAP BAPIs).
- Interfaces for all calculation strategies to support testability and future replacement.
- Factory patterns for object instantiation to ease unit testing and mocking.
- Exception classes (`ZCX_RE_IFRS16_*`) for all error handling — no message-based error handling in business logic classes.
- Unit test classes (`LTCL_*`) for all calculation logic — minimum 80% coverage target for calculation engine.

## Logging, Auditability, Performance, Security

### Logging
- All IFRS 16 calculation runs must be logged to a Z application log table with: run ID, user, timestamp, contract reference, input snapshot, output snapshot, calculation version, and status.
- Use SAP Application Log (SLG1/`BAL_*`) as the framework — **to be confirmed** whether a custom log table is also needed for long-term retention.
- Errors must be logged at the item level — partial failures must not silently skip contracts.

### Auditability
- Every posted FI document generated by the addon must carry a reference to the calculation run ID.
- Every accounting judgment stored in Z tables must carry: who entered it, when, what was the previous value (change document or history table).
- The "why" of every calculation (which rule, which parameter, which IFRS 16 paragraph) must be accessible from the calculation result.

### Performance
- Batch calculation jobs must be parallelized using `CALL FUNCTION ... STARTING NEW TASK` or work process parallelization — architecture to be confirmed with ABAP Architect.
- No full table scans on large Z tables without index-aware query design.
- Selection screen variants must be pre-configured for typical close runs to minimize user error.

### Security
- All IFRS 16 transactions must have dedicated authorization objects (`ZRE_IFRS16_AUTH_*`) — no reliance on general RE authorization alone.
- Segregation of duties: data capture role, calculation execution role, approval role, and disclosure role must be separate.
- No calculation results may be overridden without a documented reason and supervisor approval (enforced in the workflow).

## Documentation Approach
- **Living documentation:** `docs/` folder is updated with every significant change — not deferred.
- **Spec-first:** No Z object is created without a completed spec in `specs/` and a design section in `docs/technical/`.
- **Version-stamped:** Every document carries a version header. Changes are tracked in `docs/governance/decision-log.md`.
- Diagrams are maintained in draw.io or Mermaid format (embedded in Markdown) — not in proprietary tools without export.

## AI Assistant Strategy
The end-user AI layer will operate as follows:

- **Kiro agents** (this workspace): Used by the project team for design, spec writing, governance, and knowledge curation. Not exposed to end users.
- **Microsoft Copilot or Google Gemini** _(to be confirmed — depends on client IT landscape)_: Used as the end-user conversational assistant embedded in the SAP UI or connected via a side-panel. This layer guides users through data capture, explains calculation results, and answers IFRS 16 questions.
- The end-user AI layer must be trained on the same RAG knowledge base used by the project agents — to be designed as a shared knowledge source.
- **Separation of concerns:** The Kiro agent layer governs project delivery. The end-user AI layer governs runtime user experience. They do not share live state.

**Note:** All external AI integrations (Copilot, Gemini) must be formally confirmed with the client's IT security and data governance team before implementation begins.
